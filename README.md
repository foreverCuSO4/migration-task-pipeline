# Migration Task Pipeline

Migration Task Pipeline is a repository for building an agent-driven data synthesis pipeline around accelerator migration tasks.

The near-term goal is conservative: build a reliable benchmark task factory. The longer-term goal is larger: use the same task construction process to generate high-quality engineering trajectories for RL, distillation, and model training.

## What This Repository Is For

This repository will manage the pipeline that turns real upstream software projects into candidate G4 tasks:

1. Collect repository seeds from trusted software ecosystems.
2. Filter repositories by metadata, license, activity, and migration surface.
3. Scan selected repositories for CUDA/NVIDIA assumptions and taskability signals.
4. Ask agents to produce structured candidate cards.
5. Select a small number of candidates for manual pilot task construction.
6. Gradually automate the repeated parts of task construction, validation, and review.

The first practical milestone is not full automation. The first milestone is a high-quality list of candidate repositories and a repeatable way to explain why each one is or is not worth turning into a G4 task.

## G4 Working Definition

In this project, a G4 task is contract-backed rather than oracle-backed.

A G4 task starts from a real upstream NVIDIA/CUDA-oriented project and defines correctness through:

- upstream source and provenance,
- public or stable external interfaces,
- verifier-controlled inputs,
- offline references where possible,
- runtime evidence that target execution actually uses Ascend NPU,
- case-level scoring that supports partial credit.

G4 tasks do not require an existing Ascend-native reference implementation.

## Initial Pipeline Shape

The first version of the pipeline should stay simple:

```text
seed collection
  -> metadata filtering
  -> repository download or archive sampling
  -> static migration-surface scan
  -> agent candidate-card review
  -> human shortlist
  -> manual pilot task construction
```

Agents should not be used to wander GitHub blindly. Scripts should collect and filter repository candidates first. Agents should be used where judgment is valuable: reading a candidate project and deciding whether it can become a good G4 task.

## First Seed Sources

The first seed collector should focus on package ecosystems because they are likely to contain real software rather than toy repositories:

- PyPI packages in ML, scientific computing, HPC, and PyTorch ecosystems.
- conda-forge package metadata and feedstock links.
- Curated package or project lists as a secondary source.

For each package, extract GitHub repository links from metadata such as homepage, project URLs, dev URLs, source URLs, summaries, and descriptions.

## Current Milestones

### Milestone 0: Seed Collector

Produce the first repository seed table from PyPI and conda-forge.

Expected output:

```text
data/processed/repo_seeds.csv
```

Minimum columns:

```text
source,package_name,package_version,repo_url,homepage,summary,keywords,license,downloads_30d,collected_at
```

### Milestone 1: Static Scanner

Download or shallow-clone a filtered subset of repositories and scan for CUDA/NVIDIA migration signals:

- `torch.cuda`
- `.cuda()`
- `device="cuda"` and `cuda:`
- `torch.backends.cudnn`
- `torch.cuda.amp`
- `torch.distributed` with `nccl`
- `nvidia-smi`
- `cupy`
- `numba.cuda`
- `triton`
- `CUDAExtension`
- `.cu` and `.cuh`
- `nvcc`

### Milestone 2: Candidate Cards

For the top ranked repositories, run agent review jobs that output structured candidate cards.

Each candidate card should answer:

- What does the project do?
- What CUDA/NVIDIA assumptions does it contain?
- What public interface could define a G4 task?
- Can we build an offline verifier?
- What hidden cases might separate weak and strong agents?
- What are the biggest risks?
- Should we pilot this manually?

### Milestone 3: Manual Pilot

Select 2-3 candidates and manually build G4 tasks from them. The purpose is to learn what should be automated next, not to prove the entire pipeline is already automatic.

## Non-Goals For Now

- Full automatic G4 task generation.
- Automatic admission to a benchmark suite.
- Large-scale repository cloning before metadata filtering.
- Agent-driven open-ended GitHub exploration.
- Using unverified agent judgments as final selection decisions.

## Development Notes

This repository is intentionally small at the start. Prefer explicit data files, simple scripts, and inspectable intermediate outputs over a clever framework.

The pipeline should make it easy to answer:

- Where did a repository candidate come from?
- Why was it kept or rejected?
- What evidence supports its migration surface?
- What would the first manual task-construction probe be?
