# File Management Rules

This document defines the initial file management conventions for Migration Task Pipeline.

The goal is to keep the pipeline auditable. Every generated table, candidate card, and manual decision should be traceable back to its source.

## Top-Level Directories

### `configs/`

Configuration files for repeatable runs.

Examples:

- seed source lists
- keyword lists
- scanner rules
- agent model profiles
- batch run definitions

Configuration files should be small, reviewed, and committed.

### `data/`

Local pipeline data.

Use these subdirectories:

- `data/raw/`: raw downloaded metadata or source records.
- `data/interim/`: normalized but not final intermediate files.
- `data/processed/`: stable outputs consumed by later pipeline stages.
- `data/external/`: third-party files that are not generated by this project.

Large files should not be committed by default. Commit only small samples, schemas, and curated registries.

### `src/`

Importable Python package code for collectors, normalizers, scanners, scoring, and shared helpers.

Keep one-off scripts out of `src/` unless they are reusable.

### `scripts/`

Runnable CLI wrappers and operational scripts.

Scripts should call reusable code from `src/` when possible.

### `templates/`

Templates for:

- candidate-card prompts
- candidate-card YAML/JSON schemas
- reviewer reports
- task construction checklists

### `candidate_cards/`

Agent-generated candidate cards.

Recommended layout:

```text
candidate_cards/
  YYYYMMDD-run-name/
    owner__repo.yaml
```

Generated cards may be numerous. Commit curated subsets or run manifests first; decide later whether to version all cards.

### `registry/`

Human-reviewed decisions and curated lists.

Examples:

- accepted pilot candidates
- rejected candidates with reasons
- known-bad repositories
- source allowlists
- source blocklists

Registry files should be committed.

### `runs/`

Run logs and local artifacts.

This directory is ignored by git by default. Store reproducible outputs in `data/processed/` or `registry/` once reviewed.

### `notebooks/`

Exploratory notebooks only.

Do not make notebooks the source of truth for the pipeline.

## Naming Conventions

Use lowercase filenames with hyphens or underscores.

Good:

```text
repo-seed-sources.yaml
pypi_repo_seeds.csv
candidate-card.schema.yaml
```

Avoid:

```text
FinalResults2.csv
new_test.py
tmp.md
```

## Data File Conventions

Prefer CSV for flat tables and JSONL for large record streams.

Recommended files:

```text
data/raw/pypi-packages-YYYYMMDD.jsonl
data/raw/conda-forge-repodata-YYYYMMDD.jsonl
data/interim/repo-urls-normalized-YYYYMMDD.csv
data/processed/repo-seeds-YYYYMMDD.csv
```

Each processed dataset should have:

- source name,
- collection time,
- normalization version,
- enough fields to reproduce or audit the decision.

## Repository URL Normalization

Normalize all GitHub URLs to:

```text
https://github.com/<owner>/<repo>
```

Strip:

- `.git`
- `/issues`
- `/pulls`
- `/tree/...`
- `/blob/...`
- fragments and query strings

Deduplicate by lowercase `owner/repo`, but preserve original casing in display fields if needed.

## Commit Policy

Commit:

- source code,
- configs,
- schemas,
- documentation,
- small curated registries,
- small sample data needed for tests.

Do not commit by default:

- cloned repositories,
- large raw dumps,
- large run logs,
- model/checkpoint files,
- full agent traces.

If a large artifact becomes important, add an explicit artifact policy before committing it.

## Manual Review Notes

Human decisions should be written down in `registry/`, not left only in chat logs.

A rejected candidate is still useful if the rejection reason is clear:

```yaml
repo: owner/name
decision: reject
reason: "No offline verifier path; requires private dataset."
reviewed_at: 2026-06-16
```

This prevents the pipeline from rediscovering the same bad candidates.
